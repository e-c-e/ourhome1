# 我们的空间开发文档

## 1. 项目定位

### 4.4 记录今天提醒设想

这部分先只做最小范围：围绕“记录今天”这个入口，在两人空间内实现一人发布、另一人收到提醒。

#### 4.4.1 目标范围

- 只针对 `记录今天` 入口触发提醒，不扩展到纪念日、心愿、宠物岛等其他模块。
- 空间仍然严格限定为两位成员，提醒对象只在这两个人之间流转。
- 规范明确禁止第三个人进入空间，第三个 `openid` 必须在登录阶段被拦截，不允许通过任何提醒或内容入口绕过。
- 默认规则是：一方发布今天的记录后，只通知另一方。
- 先做最小可用链路，保证提醒可靠，再考虑更复杂的定时和重复能力。

#### 4.4.2 当前核心场景

| 场景 | 说明 |
| --- | --- |
| 记录今天后提醒对方 | 一个人发布今天的帖子后，只推给另一位成员，让对方及时看到 |
| 自己发布自己不提醒 | 发布人自己不必收到重复提醒，避免打扰 |
| 仅站内提醒 | 如果订阅消息还没接通，先在应用内显示提醒卡片或提醒入口 |

#### 4.4.3 规则约定

- 提醒仅由“记录今天”这个发布动作触发。
- 默认只通知另一位成员，不做双发、不做群发。
- 如果后续要扩展，也应先保持一对一语义，再考虑“自己可见”或“双方都提醒”的增强选项。

#### 4.4.4 开发落点建议

- 发布“记录今天”时，在云端记录一条提醒事件，目标用户固定为另一位成员。
- 提醒对象的确定方式很简单：先通过当前登录用户的 `openid` 找到自己，再从 `users` 集合里取 `openid` 不同的那位成员作为接收人。
- 如果当前用户的 `role` 是 `owner`，则接收人就是 `partner`；如果当前用户是 `partner`，则接收人就是 `owner`。
- 前端先预留提醒状态位，例如“已提醒 / 待提醒 / 已读”。
- 如果真实订阅消息暂时不可用，可以先做站内提醒，保证流程闭环。
- 后续若要接云端定时任务，也只针对这一个入口扩展，不要把提醒系统做成通用广播框架。

### 4.4 双人提醒系统设想

这部分是新的产品创意，目标不是做“群通知”，而是做一个只适用于两个人的小范围提醒系统。

#### 4.4.1 产品目标

- 空间只有两位成员，所有提醒都围绕这两个人展开。
- 允许一方发记录后，只通知另一方，避免双方都被重复打扰。
- 允许把提醒分成“只提醒对方”“只提醒自己”“双方都提醒”三种模式。
- 允许用户为某些内容设置定时提醒，例如指定时间、指定提前天数、重复提醒。

#### 4.4.2 核心场景

| 场景 | 说明 |
| --- | --- |
| 发记录提醒对方 | 一个人发布今天的帖子后，只推给另一位成员，让对方及时看到 |
| 定时提醒自己 | 某些内容先保存，到了设定时间只提醒自己，避免遗漏 |
| 双方共同提醒 | 适合纪念日、约定事项、共同待办，保证两个人都能收到 |
| 提前提醒 | 在纪念日前 1 天、3 天、7 天等节点自动提醒 |
| 反复提醒 | 对重要事件可以设置多次提醒，直到被查看或完成 |

#### 4.4.3 提醒类型建议

- 发布提醒：发布回忆、纪念日、心愿变化后触发。
- 定时提醒：用户指定某个时间点自动触发。
- 提前提醒：基于纪念日或计划日自动倒计时提醒。
- 状态提醒：例如心愿完成、心愿存入、纪念日临近等状态变化提醒。

#### 4.4.4 提醒对象规则

- `提醒对方`：只发给另一位成员，适合“今天发生了什么”这类私人分享。
- `提醒自己`：只发给当前操作人，适合记录备忘、定时待办。
- `提醒双方`：两个人都收到，适合共同纪念日或共同目标。
- 默认规则建议按内容类型区分：
	- 回忆记录默认提醒对方。
	- 个人草稿和备忘默认提醒自己。
	- 纪念日和共同计划默认提醒双方。

#### 4.4.5 开发落点建议

- 前端需要补一个“提醒对象”和“提醒时间”的设置区。
- 数据层需要给记录增加提醒配置字段，例如提醒对象、提醒时间、重复规则、是否已发送。
- 云端需要一个真正可执行的提醒链路，后续可以接订阅消息或定时触发器。
- 如果短期还没有真实推送能力，可以先做站内提醒和提醒列表，保证逻辑闭环。
- 由于当前空间严格只有两个人，提醒路由不需要群发逻辑，只要根据双方身份做一对一分发即可。

#### 4.4.6 后续演进方向

- 先做“站内提醒 + 提醒列表”，再接真实订阅消息。
- 再补“定时任务 + 触发器”，把提醒从页面内动作变成云端自动执行。
- 最后可以把提醒和纪念日、回忆、心愿串起来，形成一个统一的双人时间线。

## 4.5 UI 细节规范

- 表单输入框和主操作按钮之间必须保留明确的纵向呼吸感，避免紧贴导致界面发闷。
- 涉及多个表单项之间的间距、分组上下关系、按钮与输入区之间的布局时，优先由页面级容器统一控制，例如使用外层 `flex + gap`、独立包裹层或明确的块级间距；不要把页面排版依赖在组件内部类名上，否则容易出现“代码已修改但界面无变化”的假象。
- 使用自定义组件（如 `warm-input`、`warm-button`）时，若视觉调整未生效，必须先确认样式究竟作用在页面宿主、组件根节点还是组件内部节点；布局问题先查组件边界，再决定改页面容器还是改组件实现，不允许直接假设页面样式一定能穿透组件内部结构。
- 进度条、说明文本和删除/次操作按钮之间要避免过宽留白，优先通过更紧凑的排版关系提升整体感。
- 像心愿花园这类情绪化页面，布局尽量做成多行多列、有层次的场景，不要只停留在普通信息卡片堆叠。
- 底部双操作按钮优先使用对称布局，主次关系通过色彩和通透度区分，不要依赖不等宽造成失衡。
- 悬浮在页面内容之上、卡片之上或场景中的按钮，统一使用“半透明糖玻璃胶囊按钮”这套样式，不再退回普通纯色矩形按钮。
- 这套悬浮按钮样式的最低要求是：圆润胶囊外轮廓、半透明或玻璃感材质、柔和顶部高光、细腻渐变过渡、主次按钮通过透明度和发光强弱区分，而不是只通过大小失衡或生硬色块区分。
- 悬浮主按钮应保持柔和发光感和更明确的视觉聚焦，但不能出现高光过硬、颜色过满、像实体塑料块一样压在页面上的问题；悬浮次按钮应更轻、更透，接近半透明玻璃质感。
- 固定在页面底部的操作区必须提前为滚动内容预留足够底部空间，避免最后一组表单项、选项或说明文字被遮挡。
- 固定按钮不得直接复用悬浮糖玻璃样式，必须使用另一套更稳定、更贴地的固定操作样式，避免页面底部像飘着两块糖玻璃。
- 固定按钮的推荐形态是“托盘 + 按钮本体”两层结构：外层托盘负责承接页面底部并与内容收口，内层按钮负责主次操作表达；托盘允许轻微半透明，但整体必须比悬浮按钮更实、更稳。
- 固定按钮的主按钮应采用更高密度的暖粉或暖红渐变，透明度明显低于悬浮按钮，文字对比更强，高光更窄更收敛，避免继续做成整块通透发亮的糖玻璃。
- 固定按钮的次按钮应采用奶白、雾粉或浅米色实体感底面，依靠清晰边界、细描边和弱阴影建立层次，不再靠大面积透底来区分主次。
- 固定按钮的圆角仍应柔和，但整体厚度感、边界感和按压反馈必须更明确；可保留轻微顶部柔光与细腻渐变，不可保留大面积悬浮泛光、强透底或过度背景渗透。
- 若按钮固定在页面底部，外层承托区域应使用偏低饱和、偏浅暖色的结构托盘，并通过圆角、上缘柔光、下缘投影完成“固定、承接、收口”的作用；不要继续套用悬浮按钮那种弱托盘、强漂浮的处理方式。
- 固定操作区内若存在双按钮，优先采用等宽对称布局；主按钮负责聚焦，次按钮负责平衡，但两者都应落在同一托盘体系内，不要像两个独立漂浮物。
- 后续新页面或改版页面涉及操作按钮时，必须先判断按钮属于“悬浮按钮”还是“固定按钮”：悬浮按钮沿用当前这套半透明糖玻璃语言，固定按钮单独设计，不得混用。

## 4.6 刷新体验优化规范

- 页面首次进入、主动下拉刷新、跨模块返回后需要重新校验核心数据时，允许整页拉取并展示 `loading`。
- 页面内的单条操作，例如心愿摘取收藏、挂回树上、切换完成状态、单条删除、新增一条内容，默认不允许再走“整页重新请求 + 整页 loading”的链路。
- 这类单条操作优先使用“本地状态局部更新 + 云端同步”的方式处理，只改受影响的列表项、统计值和文案，不重建整页结构。
- 如果服务端返回失败，再做局部回滚或补拉一次数据；失败兜底不能反过来成为所有正常操作都整页刷新的理由。
- 涉及列表迁移的交互，例如“树上果实移入收获匣”这类动作，必须优先保证视觉上是元素位置变化，而不是整页重新出现。
- 月份切换、筛选条件切换这类确实会改变整块数据集的动作，可以保留分区刷新，但也应尽量避免让不相关区域一起闪动。
- 后续页面改造优先级按体感排序：心愿树页、首页返回链路、时光墙/日历页、纪念日页、设置类页面。

### 4.6.1 本轮落地

- 心愿树页已作为第一批改造对象：新增、完成、摘下收藏、挂回树上、移除，改为前端局部更新，不再在成功后整页调用 `loadPageData`。
