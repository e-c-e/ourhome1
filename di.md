# 我们的空间开发文档

## 1. 项目定位

“我们的空间”是一个严格双人使用的微信小程序，主题不是做泛社交，也不是做重游戏化养成，而是围绕“两个人的日常记录、纪念、回看和轻提醒”展开。

当前产品重心已经从“内容瀑布首页”调整为“今日状态首页 + 时光墙主浏览 + 纪念日主整理 + 心愿/宠物作补充”。

核心原则：

- 空间只允许两位成员进入，第三个账号必须被拦截。
- 首页负责展示当下状态和下一步动作，不再承担完整内容流浏览。
- 时光墙负责连续浏览图片和回忆正文。
- 纪念日负责重要日期维护、年度进度、提醒设定。
- 提醒系统当前先保证“记录今天”闭环，纪念日提醒还未接成真正的自动通知链路。

## 3. 技术与架构现状

### 3.1 技术栈

- 微信小程序原生页面：WXML + LESS + JS
- 云开发数据库
- 云函数：`auth`、`relationship`
- UI 基础：项目内自定义暖色组件 + TDesign Mini Program

### 3.2 核心云函数职责

#### `auth`

- 负责双人登录授权
- 初始化空间默认结构
- 拦截第三人进入
- 新用户写入基础字段和订阅状态

#### `relationship`

- 首页聚合数据 `getHomeData`
- 记录今天 `createMoment`
- 点赞、评论、删除、详情
- 纪念日、心愿单、宠物岛、日历日记数据处理
- 站内提醒和微信订阅提醒相关接口
- 首页活动流与小程序进入记录

### 3.3 关键数据集合

| 集合 | 用途 |
| --- | --- |
| `users` | 双人账号、角色、订阅状态、最近一次进入小程序信息 |
| `space` | 空间资料、在一起日期、首页小提示、提醒设置、宠物、活动流 |
| `moments` | 记录今天 / 时光墙主内容 |
| `anniversaries` | 自定义纪念日列表 |
| `wishes` | 心愿单数据 |
| `notifications` | 站内提醒记录 |

## 4. 首页现状与设计思路

首页已经不再展示“最近回忆”轮播，而是围绕“今天适合关注什么”组织内容。

### 4.1 首页结构

当前首页由以下区域组成：

1. Hero 区：在一起天数、回忆条数、最近心情、快捷记录入口
2. 今日状态四卡：
	 - 今天谁何时来过
	 - 今天的小提示
	 - 纪念日倒数 / 提醒卡
	 - 对方最近动态
3. 提醒中心：站内提醒列表 + 微信提醒授权入口

### 4.2 今日状态四卡的实际规则

#### 今天谁何时来过

- 记录的是“谁何时进入这个小程序”，不是“谁何时打开首页”。
- 入口记录点已放到 `App.onShow`，避免首页成为统计入口。
- 如果今天来过，显示今天进入时间。
- 如果今天没来但昨天来过，显示昨天进入时间。
- 如果更早来过，显示最近一次进入时间。
- “进入小程序”信息已经从完整动态页剥离，统一回到这一张卡。

#### 今天的小提示

- 默认有系统内置提示池。
- 首页支持直接编辑。
- 空间设置页也保留编辑入口。
- 一行一条提示，按日期轮换显示。
- 留空时回退到系统默认提示。

#### 纪念日倒数 / 提醒卡

- 首页这张卡当前不是单独的“提醒系统”，而是首页状态摘要。
- 逻辑以最近的纪念日为准，来源包括：
	- `space.startDate` 对应的在一起纪念日
	- `anniversaries` 集合里的自定义纪念日
- 当前策略：
	- 如果最近纪念日还大于 30 天：显示“下一次重要节点是 xxx”，告诉用户还有多少天，进入 30 天内才开始提醒语气
	- 如果已经进入 30 天内：显示“还有 x 天到 xxx”
	- 如果当天：显示“今天就是 xxx”
- 纪念日页数据变动后，会给首页打 `HOME_DIRTY`，保证切回首页时静默更新，不影响其他板块。

#### 对方最近动态

- 基于活动流摘要，而不是临时遍历 moments 拼接。
- 当前展示对象包括：
	- 对方发布记录
	- 对方评论
	- 对方点赞
- 不再展示“进入小程序”动作，这部分已移回来过板块。
- 首页卡只显示最近摘要和少量条目，完整内容进入“最近动态”页查看。

### 4.3 首页刷新策略

首页不是每次 `onShow` 都整页重拉，而是采用脏标记策略：

- `HOME_DIRTY` 存在时：首页回到前台会静默刷新
- 没有脏标记时：保留当前数据，避免频繁抖动
- 当前已经会触发首页刷新脏标记的场景：
	- 发布记录
	- 动态详情中的点赞、评论、删除
	- 纪念日页修改在一起日期
	- 新增纪念日
	- 删除纪念日

这套策略的目标是：

- 让首页状态块保持最新
- 同时不拖累墙、日历、宠物等其他页面跟着一起刷新

## 5. 最近动态系统现状

### 5.1 当前方案

- 活动流存储在 `space.activityFeed`
- 每次关键动作发生时，向活动流 append 一条轻量事件
- 首页最近动态卡与完整动态页共用同一份活动流数据

### 5.2 已纳入活动流的动作

- 发布记录
- 点赞
- 评论
- 进入小程序

注意：

- 当前完整动态页已经过滤掉“进入小程序”动作
- 活动流里仍保留这类数据，供首页“今天谁何时来过”板块使用

### 5.3 完整动态页的职责

- 作为“最近动态”的完整时间线
- 当前只保留内容型动态：记录、点赞、评论
- 若动态关联了 `momentId`，允许跳转查看详情

### 5.4 后续建议

- 增加“今天 / 昨天 / 更早”分组
- 增加筛选：只看对方 / 只看评论 / 只看点赞
- 增加今天来过次数统计，回到首页来过卡里展示

## 6. 记录今天提醒链路现状

### 6.1 已闭环部分

当前已经真正闭环的是“记录今天”提醒：

1. 用户从发布页提交记录
2. 前端走 `createMoment`
3. 云函数 `relationship.createMoment` 创建记录
4. 云端创建一条站内提醒给另一位成员
5. 如果对方开通过微信提醒授权，则尝试下发订阅消息
6. 首页提醒中心展示站内提醒，并支持标记已读

### 6.2 当前规则

- 严格只通知另一方，不通知自己
- 空间严格双人，不做多人广播
- 站内提醒优先，微信订阅提醒作为增强能力

## 7. 纪念日模块现状

### 7.1 已实现部分

- 在一起日期维护
- 自定义纪念日新增 / 删除
- 年周期进度展示
- 100 天里程碑进度
- 提醒设置保存：
	- `enabled`
	- `daysBefore`
	- `inAppEnabled`
	- `serviceEnabled`
- 纪念日页内的提醒提示生成

### 7.2 当前提醒机制的真实状态

纪念日模块目前有“提醒设置”，但还没有真正的自动提醒派发闭环。

当前已经有的，是：

- 页面内可见的提醒提示
- 提前天数切换
- 开关状态保存到 `space.reminderSettings`

当前还没有的，是：

- 到日期前自动生成站内通知
- 自动下发纪念日微信订阅消息
- 去重规则，例如“同一天同一个纪念日不重复提醒”
- 云端定时任务

### 7.3 首页纪念日卡与纪念日提醒设置的区别

这是当前最容易混淆的点：

- 首页纪念日卡：首页状态摘要，负责告诉用户“最近的重要日子还有多久”
- 纪念日页提醒设置：纪念日模块自己的提醒策略配置

两者目前不是同一条链路。

当前首页纪念日卡已经实现了“30 天内开始提醒语气”，但它不等于纪念日页已经接通自动通知。

## 8. 刷新体验优化规范

### 8.1 总原则

- 首次进入、主动下拉、核心数据重新校验，允许整页加载
- 单条操作默认不允许触发整页 loading
- 单条动作优先使用“本地状态局部更新 + 云端同步 + 必要时脏标记”

### 8.2 已落地页面

- 心愿单：新增、完成、收藏、恢复、删除已改为局部更新
- 时光墙 / 动态详情：点赞、评论、删除后通过脏标记驱动首页 / 墙 / 日历刷新
- 首页：依赖 `HOME_DIRTY` 做静默刷新
- 纪念日：改动后只打首页脏标记，不扩大刷新范围

### 8.3 当前脏标记

| 标记 | 用途 |
| --- | --- |
| `HOME_DIRTY` | 首页状态和摘要更新 |
| `WALL_DIRTY` | 时光墙内容需要刷新 |
| `DIARY_DIRTY` | 日历日记需要刷新 |

## 9. UI 资产与已验证模式

### 9.1 已沉淀的“最近回忆”轮播模式

虽然首页已经移除“最近回忆”模块，但横向主卡聚焦轮播已经验证可复用，适合作为后续精选内容模块的 UI 资产。

#### UI 方法

- 使用原生 `swiper`
- 通过 `previous-margin + next-margin` 做主卡居中
- 主卡和侧卡用 `scale + opacity + saturate` 拉开层次
- 图片 `aspectFill`
- 无图时使用统一文字占位卡

#### 已验证参数

- `swiper.previous-margin`: `72rpx`
- `swiper.next-margin`: `72rpx`
- `swiper.duration`: `560`
- `swiper.easing-function`: `easeInOutCubic`
- 主卡：`scale(1)`、`opacity: 1`、`saturate(1.02)`
- 侧卡：`scale(0.92)`、`opacity: 0.56`、`saturate(0.9)`
- 卡片最大宽度：`588rpx`
- 媒体高度：`268rpx`

### 9.2 当前按钮与布局规范

- 浮层按钮和固定底部按钮不混用一套视觉语言
- 固定操作区必须预留底部安全空间
- 多表单页面优先用外层容器控制节奏，不把布局寄托在组件内部样式穿透上
- 情绪化页面优先做场景层次，不要只堆信息卡

## 11. 重新梳理开发思路

如果接下来要继续开发，建议按下面顺序，不要再同时铺太多面：

### 第一阶段：把提醒边界彻底理顺

先明确三类提醒不是一回事：

1. 首页状态摘要
2. 页面内提示
3. 真正的站内 / 微信提醒

当前“记录今天”已闭环，纪念日未闭环。后续所有提醒设计都要先判断属于哪一层，不要再混成一个概念。

### 第二阶段：补纪念日自动提醒闭环

建议按这个顺序实现：

1. 到日期前 `N` 天自动生成站内提醒
2. 同一天 / 同纪念日去重
3. 已授权用户再尝试微信订阅提醒
4. 最后才考虑更复杂的定时任务 UI

### 第三阶段：完善首页来过与动态信息

建议补：

1. 今天进入小程序次数
2. 最近一次进入页面
3. 最近动态页分组与筛选

### 第四阶段：保持刷新策略收敛

后续新增功能仍然遵守：

- 单页局部更新优先
- 用脏标记驱动跨页同步
- 不允许为了“稳”回退成整页重新加载

## 12. 当前验收重点

1. 双人授权是否稳定，第三人是否被正确拦截
2. 首页今日状态四卡是否符合现有规则
3. 发布记录后，首页提醒中心和对方提醒是否正常
4. 纪念日修改后，切回首页是否能静默更新
5. 30 天内纪念日卡是否进入提醒语气，30 天外是否回退为普通提示
6. 最近动态页是否只显示记录、点赞、评论，不再显示进入小程序
7. 首页“今天谁何时来过”是否能正确展示最近进入信息

## 13. 当前必须重新部署的云函数

每次涉及以下能力修改后，都应优先重新部署云函数：

1. `relationship`
2. `auth`

尤其是以下功能如果出现“页面表现和代码不一致”，优先先排查是否未重新部署：

- 首页纪念日卡文案
- 首页来过信息
- 最近动态活动流
- 记录今天提醒
- 登录拦截与默认字段
